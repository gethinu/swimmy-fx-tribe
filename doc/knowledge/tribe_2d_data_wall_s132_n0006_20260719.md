# tribe-2d — data-wall follow-up: s132 (GBPJPY) & n0006 (EURCHF) faithful D1 import (measured 2026-07-19)

**Question (owner).** The prior run (`tribe_2d_bundle_founder_import_20260719.md`, commit `1db94384`) found the
real bundle non-USDJPY edges are almost all un-importable — mechanism outside the tribe genome/scorer, or a
symbol with no tribe data. Two edges were the *one clean* exception: **cleanly representable by a tribe
primitive, blocked only by missing data** — **s132 = GBPJPY Donchian-7 D1** and **n0006 = EURCHF BB-20 D1**.
This run removes that data wall (feeds the edges' **real canonical D1 series** to the tribe's **own** honest
CPCV scorer, on the **real symbols** — not the prior xplant onto EURUSD/GBPUSD/EURJPY) and settles, for each
edge, the honest 3-way question:

- **(i)** data in, PF/Sharpe **and CPCV** substantially robust, fails **only** the density floor → *mechanism
  real, tribe density bar too strict for D1 rare-events* (policy call).
- **(ii)** data in, PF/Sharpe/**CPCV itself** non-robust (negative / straddles zero) → *mechanism non-robust,
  not a data problem* (settled).
- **(A)** data in, passes tribe-robust outright → *owner hypothesis "data is the only wall" revived.*

**Verdict up front (measured): both s132 and n0006 → (ii).** With the data fully in place and the real
symbols, both edges' **CPCV fails on substance, not density** (0/10 folds density-starved; 7/10 folds are
genuine losers). A single-window OOS view *masquerades* as (i) — both are positive on 2021-2025 (PF ~1.5,
pen-Sharpe ~0.5) and fail only the ≥200-trade count there — but the full-history CPCV, the actual robustness
instrument, shows both work in only ~3/10 regimes. Removing all friction does not rescue either. **"試して
落ちた," decisively — and the thing it fell on is the mechanism, not the missing data.**

Guardrails honored: redirect harness only (standalone `primitive_scan.exe`, no repo-root `run-all-tests`);
live `swimmy.db` never touched (no DB pipeline run); `mt5_Bundle` read-only; **no tracked source modified**

## 1. Data availability — OBTAINED (this is the crux vs "couldn't try")

No GBPJPY or EURCHF **M1** exists anywhere (bundle `data/ohlc_m1/` has neither; parquet cache stops at M15).
But s132/n0006 are **D1 edges validated by the bundle on D1 bars** — their canonical series is the MT5 broker
**D1** CSV, which *is* present and is exactly what the bundle used:

| edge | bundle canonical series (read-only) | bars | span | tribe-OOS[2021-2025) bars |
|---|---|---|---|---|
| s132 | `data/ohlc_d1/gbpjpy_d1_mt5_20030101_20260713.csv` | 6115 | 2003-01-01 .. 2026-07-14 (23.5y) | 1040 |
| n0006 | `data/ohlc_d1/eurchf_d1_mt5_20030101_20260508.csv` | 6057 | 2003-01-01 .. 2026-05-08 (23.3y) | 1041 |

Converted to tribe format (`timestamp,open,high,low,close,volume`, Unix-sec, `convert_d1.py`) — strict-
monotonic, 0 dups. **Faithfulness of feeding D1 to an M1-oriented scorer:** `run_on()` resamples the input to
`tf_seconds` before backtesting; with `tf_seconds=86400` and day-aligned D1 timestamps, each bar floor-divides
to its own bucket, so `resample_candles(D1, 86400)` is the identity — the scorer backtests on the **exact
canonical D1 bars**. Confirmed: this is the same series, not apples-to-oranges.

**Representability caveat (honest scope).** The tribe primitive represents the *core* only. It cannot represent
s132's EMA50/200 spread gate or ATR trailing stop, nor n0006's SMA20/200 filter or short-only bias. So what is
tested is *"does the tribe-representable core of each edge, on its real symbol/data, pass the tribe honest
gate."* That is the maximal faithful test available — and it is the relevant one, because the tribe could only
ever ingest the core.

## 2. The tribe honest gate (floor NOT lowered)

`guardian/src/bin/primitive_scan.rs` (the verification engine; rebuilt from clean source this run). OOS window
**2021-01-01 .. 2025-01-01**; CPCV = full input range, 10 equal-index blocks (~2.35y each). Gates:
`OOS trades≥200, PF≥1.10, pen-Sharpe≥0.30`; `CPCV pass-rate≥0.60 with ≥20 trades/fold & PF≥1.10 & pen≥0.30`;
`CPCV median-sharpe<2.0`. "tribe-CPCV-robust" ≡ oos_qualified **and** cpcv_ok. Barriers faithful to the EAs:
`atr` mode, `sl=2.0` (= `InpHardSLATR=2.0×ATR`), s132 `max_hold=40`, n0006 `max_hold=20`; primary uses a
non-binding `tp=99` (exit governed by Donchian channel / BB-mid + SL + max_hold, mirroring the EA); `tp=3` and
`max_hold=0` run as sensitivity. Cost run at each symbol's bundle RT (`GBPJPY 2.5p → slip 0.0125`;
`EURCHF 2.0p → slip 0.0001`) and **frictionless (slip 0)**.

  and answers no; positive pooled EP ≠ CPCV-robust.**

is the wall. Honest disclosure: a naive recent-window-only read (positive PF+Sharpe, density-blocked) would
have called this (i). The full-history CPCV is what makes it (ii).

## 6. Owner-hypothesis final verdict

**"Data is the only wall; stack GBPJPY/EURCHF and the path to diversification opens" — NOT revived.** For the
two cleanest data-blocked candidates, once the real data and real symbols are supplied, the tribe-representable
mechanisms are **non-robust across regimes on substance** (CPCV 0.30, 7/10 folds losing), independent of the
density floor and of cost. Data was genuinely *not* the only wall for s132/n0006 — the mechanism is. This
tightens, not loosens, the prior conclusion: the real lever is the tribe gate / primitives / a robustness
instrument suited to rare-event edges — **and even those would still demand cross-regime robustness these two
edges do not have.** The synthetic keltner EURUSD-REVERSION seed (3 diverse-robust) remains the only thing that
has ever passed the tribe gate; the real bundle edges, data wall removed, still score 0.

## 7. Next moves (owner decision)

1. **Stop seeding the tribe from bundle D1 edges.** The two instruments (bundle pooled-EP longspan vs tribe
   per-regime CPCV) are now known to disagree exactly on regime-robustness, and the tribe CPCV is the stricter,
   correct one for "robust." No cleanly-representable bundle non-USDJPY edge survives it.
2. If D1 rare-event edges are still wanted, the change is a **robustness instrument** (small-N sealed holdout
   instead of ≥200-trade CPCV) — but this run shows that even with an N-appropriate instrument these two would
   fail, because their *per-regime* PF straddles 1.0. So this is not the cheap win.
3. Diversity, if it comes, will come from the tribe's own forward-robust region (the keltner/BB/donchian
   neighbourhoods that *do* pass CPCV on EURUSD/GBPUSD/EURJPY), not from importing the bundle book.
